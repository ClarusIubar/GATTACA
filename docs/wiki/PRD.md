# 제품 요구사항 정의서 (PRD)

## 1. 제품 비전

추억열차는 단체방에서 흘러간 약속을 장기 보관 가능한 정거장 기록으로 바꾼다. 카카오톡에서는 투표와 조율을 하고, 추억열차에서는 확정된 일정과 그날의 추억을 보관한다.

## 2. 사용자 역할

1. 게스트
   - 공개 페이지와 기록 목록을 읽을 수 있다.
   - 작성은 불가능하다.

2. 승인 대기 사용자
   - Kakao OAuth 로그인 후 생성된다.
   - 운영자 승인 전까지 작성 기능은 차단된다.

3. 승인 사용자
   - 이벤트, 메모리, 코멘트를 생성할 수 있다.
   - 본인 작성물을 수정할 수 있다.
   - 삭제는 불가능하다.

4. 운영자
   - 사용자 승인/반려를 수행한다.
   - 이벤트, 메모리, 코멘트를 삭제할 수 있다.

## 3. 기능 요구사항

### REQ-01. 확정 일정 기록

- 일정은 언제, 어디서, 무엇을, 어떻게, 확정 요약을 포함해야 한다.
- 목록은 정거장형 타임라인으로 제공한다.
- 월간 캘린더 UI는 v1 필수 요구사항이 아니다.

### REQ-02. 이벤트별 메모리 기록

- 각 이벤트 상세에서 사진과 캡션을 남길 수 있어야 한다.
- 사진은 URL 또는 업로드 저장소를 통해 연결되어야 한다.
- 메모리에는 작성자와 기록 시각이 연결되어야 한다.

### REQ-03. 코멘트 방명록

- 메모리 하위에 코멘트를 남길 수 있어야 한다.
- 승인 사용자만 작성할 수 있어야 한다.

### REQ-04. Kakao 로그인과 승인 기반 접근 통제

- Kakao OAuth로 로그인해야 한다.
- 첫 로그인 사용자는 pending으로 생성한다.
- 승인 전 작성 기능은 차단한다.
- 운영자만 승인/반려할 수 있다.

### REQ-05. 운영자 삭제 권한

- 이벤트, 메모리, 코멘트 삭제는 운영자만 가능하다.

### REQ-06. Kakao relay 경계

- 브라우저가 Kakao API를 직접 호출하지 않는다.
- Worker가 세션과 token을 기준으로 relay한다.
- 이벤트 저장은 Kakao relay 성공 여부에 의존하지 않는다.
- 이벤트 생성 직후 Kakao relay를 자동 호출하지 않는다.
- Kakao 알림은 별도 등록된 알림 채널과 명시 전송 액션이 있을 때만 보낸다.

## 4. 비기능 요구사항

- Cloudflare Pages/Workers 기반으로 배포한다.
- production에서 demo fallback을 노출하지 않는다.
- Kakao secret은 프론트엔드에 노출하지 않는다.
- live에서는 `/api/runtime-status`로 현재 배포 상태를 readback할 수 있어야 한다.
- UI/UX, 테스트, 배포, SDD 문서는 구현과 일치해야 한다.

## 5. 현재 완료 경계

- Worker 백엔드, D1 CRUD, KV 세션, R2 업로드, 서버 권한 강제, Kakao OAuth callback, Kakao relay endpoint가 구현되어 있다.
- UI는 홈/목록/상세/등록/소개 기준으로 정거장형 정보구조를 제공한다.
- 테스트 계층은 unit, integration, regression, e2e, smoke로 분리되어 있다.
- 남은 완료 증거는 PR/main 배포 후 production readback과 실제 Kakao OAuth 왕복 확인이다.
## TSK-002-16 PRD Update

- 이벤트 목록은 `언제`, `어디서`, `누가`, `무엇`, `어떻게`를 한눈에 읽을 수 있어야 한다.
- 일정 등록, 일정 수정, 메모리 등록, 메모리 수정의 시간 입력은 날짜+시+분 선택으로 제공해야 한다.
- 사용자-facing 화면에 `datetime-local` 또는 native time input을 사용하지 않는다.
- `/submit` 화면은 확정 일정 입력에 집중하며 별도 체크리스트 패널을 두지 않는다.
- 상단 내비게이션에는 `운영 원칙`/About 탭을 두지 않는다.
- 운영실은 승인/반려, 이벤트 삭제, 상세 진입, 운영 요약을 제공해야 한다.
# TSK-002-17 PRD Update

- 이벤트 상세에서 메모리는 사진 없이 캡션만으로도 생성할 수 있어야 한다.
- 사진 파일 또는 사진 URL은 선택 입력이다. 사진이 없으면 화면은 fallback 이미지를 사용한다.
- 댓글은 메모리 하위 기록이므로, 사용자가 먼저 텍스트 메모리를 남긴 뒤 그 아래에 댓글을 작성할 수 있어야 한다.
- Worker API는 `photoUrl` 누락을 저장 실패로 처리하지 않고 빈 문자열로 정규화한다.
