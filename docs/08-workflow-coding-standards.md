# 08. 개발 워크플로우와 코딩 표준

## Git 운영 방식
- 기본 브랜치: `main`
- 기능 브랜치: `codex/<feature-name>`
- 커밋 메시지 규칙:
  - `feat:`
  - `fix:`
  - `docs:`
  - `test:`

## PR / 머지 기준
- README와 관련 `docs/` 문서가 함께 갱신되어야 함
- 린트와 빌드가 통과해야 함
- 권한 변경이 있으면 RLS 문서와 UI 흐름을 같이 검토

## 코딩 표준
- 타입 정의는 재사용 가능한 공용 파일에 둔다.
- 권한 체크 로직은 컨텍스트 계층에서 일관되게 처리한다.
- UI 문구는 한국어 기준으로 유지한다.
- 삭제/승인 등 위험한 액션은 운영자 권한으로 한정한다.

## 개발 사이클 체크리스트
- 작업 시작 전:
  - 요구사항과 문서 범위를 확인했는가
  - 권한 모델에 영향이 있는가
- PR 전:
  - 테스트와 빌드가 통과하는가
  - 문서와 구현이 일치하는가
  - 불필요한 로그와 주석이 없는가

## 문서화 기준
- README에는 실행, 배포, 권한, 문서 링크를 포함한다.
- Wiki는 `docs/` 기반 요약본으로 운영한다.
- 구현 변경은 개발기록과 회고 문서에 누적한다.
